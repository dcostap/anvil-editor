# Native Decoration API Design

## Status

Draft for the native-editor replacement work. The initial native Buffer decoration store, Lua bridge, native view rendering path, and literal search result producer exist.

## Problem

The Lua-hosted `NativeEditorView` has accumulated several one-off visual features: bracket match frames, selection occurrence highlights, column/line guides, whitespace marks, indent guides, and Tree-sitter highlighting. More producers are waiting behind the same problem: search results, git diff markers, diagnostics, and line hints.

Those features should not grow into another `DocView` monkey-patch surface. The native editor needs one shared decoration model that producers can update and that the view can render consistently.

## Goals

- Give Lua and native producers a stable way to attach non-text visuals to a native Buffer or Editor view.
- Support the first real producer: persistent search result highlights.
- Leave room for git diff gutter/overview markers, diagnostics, bracket match, selection occurrences, and line hints.
- Keep text state native-owned. Decorations may reference Buffer positions, but must not expose piece-tree internals or rebuild `Doc.lines`.
- Make decoration replacement atomic per producer, so stale highlights do not flicker through partial updates.
- Keep the first implementation small enough to land before moving more render helpers out of `native_editor.lua`.

## Non-goals for the first slice

- Full renderer migration to C.
- Long-term compatibility with old `DocView` draw hooks.
- A general retained-mode UI framework.
- Perfect range remapping through every edit. The first slice may clear or recompute producer decorations after Buffer changes.
- Tree-sitter syntax highlighting replacement. Syntax spans can remain separate until a later unification pass.

## Vocabulary

- **Decoration**: a non-text visual annotation anchored to a Buffer range, Buffer line, or view coordinate.
- **Producer**: the feature that owns a set of decorations, such as `search`, `gitdiff`, `diagnostics`, or `bracket-match`.
- **Decoration set**: the current atomic output for one producer on one Buffer or view.
- **Anchor**: the position model used by a decoration: byte range, line range, or view-space geometry.
- **Plane**: where the decoration is drawn: text background, text adornment, line background, gutter, overview/scrollbar, overlay, or hint text.

## Ownership model

There should be two stores eventually:

1. **Buffer decoration store**
   - Shared by all views of the same Buffer.
   - Used for search results, diagnostics, git diff markers, bracket match if Buffer-owned, and line hints.
   - Stored in native C with the Buffer so queries can be efficient and independent of a particular Lua view.

2. **View decoration store**
   - Owned by an Editor/view instance.
   - Used for view-local guides, active selection occurrences, active IME adornments, and other per-view state.
   - May initially stay in Lua while the view shell is Lua-hosted, but should use the same decoration item shape.

The first implementation builds the Buffer store and has `NativeEditorView` query it while drawing visible lines.

## Decoration kinds

### Range highlight

Anchored by byte offsets `[start_offset, end_offset)`.

Primary uses:

- search results
- diagnostics squiggles/underlines
- bracket match ranges
- selection occurrence highlights
- inline diff ranges

Fields:

```lua
{
  kind = "range",
  start_offset = 123,
  end_offset = 128,
  plane = "background" | "underline" | "outline" | "text",
  style = "search_selection",
  priority = 100,
  flags = { active = false },
}
```

### Line marker

Anchored by zero-based Buffer lines, either a single `line` or inclusive `line_first` / `line_last`.

Primary uses:

- git inserted/modified/deleted line markers
- diagnostic line background
- test/coverage markers
- current search match line emphasis, if wanted

Fields:

```lua
{
  kind = "line",
  line = 42,
  -- or line_first = 42, line_last = 45,
  plane = "line-background" | "gutter" | "overview",
  style = "gitdiff_modification",
  priority = 100,
}
```

### Line hint

Anchored after a zero-based Buffer line. This is a later slice.

Primary uses:

- inline blame text
- diagnostic messages
- generated hints

Fields:

```lua
{
  kind = "line-hint",
  line = 42,
  text = "unused variable",
  style = "line_hint",
  priority = 100,
}
```

### View guide

View-local vertical/horizontal geometry. This is not part of the initial Buffer store.

Primary uses:

- column guides
- indent guides when generated by the view
- line guide/rulers

Fields should mirror the common decoration shape but use a view store, not Buffer byte positions.

## Planes and draw order

The renderer should group by plane rather than by producer. Producers describe what they want; the view decides where it belongs in the render stack.

Initial planes:

| Plane | Meaning | Examples |
| --- | --- | --- |
| `line-background` | Full visual line background below selections/text | diagnostics line tint, diff changed-line tint |
| `background` | Inline text-range background below glyphs | search result fill, selection occurrence fill |
| `gutter` | Marker in reserved gutter lane | git diff bar, diagnostic dot |
| `overview` | Scrollbar/overview stripe | git diff/search/diagnostic overview marks |
| `underline` | Line/curve below text glyphs | diagnostics, IME-like underline |
| `outline` | Frame around text range | active search match, bracket match |
| `text` | Optional text color/style override | future semantic overlays; lower priority than syntax unless explicit |
| `overlay` | Above text but below carets | temporary active adornments |
| `hint` | Non-buffer text anchored to a line | line hints/inlay text |

Recommended order in the Lua view while it remains the shell:

1. editor background
2. current-line and full-line background decorations
3. view guides that should sit behind text
4. inline background decorations
5. selections
6. syntax text
7. whitespace marks / inline hint text
8. underline and outline decorations
9. gutter markers / line numbers, where applicable
10. carets and IME overlay
11. scrollbar / overview markers

This can be adjusted as actual producers land, but the order should be centralized instead of producer-specific monkey patches.

## Producer lifecycle

Producers update decorations by replacing their whole set atomically:

```lua
buffer:set_decorations("search.results", decorations, {
  clear_on_edit = true,
  generation = search_generation,
})
```

The Buffer store keeps decorations grouped by `producer`. Replacing a producer set should not affect other producers. Clearing is explicit:

```lua
buffer:clear_decorations("search.results")
```

Lua-facing Buffer methods in the initial store:

```lua
buffer:set_decorations(producer, items, opts) -> true | nil, error
buffer:clear_decorations(producer) -> true
buffer:decorations(start_offset, end_offset, opts) -> items
buffer:decoration_generation() -> integer
```

Useful query options:

```lua
{
  plane = "background",      -- optional exact plane
  planes = { "background", "outline" },
  producer = "search.results",
  kind = "range",
  include_stale = false,
}
```

The query result should be sorted deterministically:

1. plane order
2. priority ascending
3. producer name
4. anchor start
5. insertion order within producer set

## Edit invalidation and tracking

The first implementation should support a conservative invalidation policy:

- Every Buffer edit increments a native decoration/change generation.
- Decoration sets record the generation where they were produced.
- Sets with `clear_on_edit = true` are removed on any edit/snap.
- Sets without `clear_on_edit` stay until their producer replaces or clears them.

This is enough for search: search results can clear on edit and be recomputed by the search UI. It is also acceptable for git diff: the git producer can recompute when its file watcher/text-change scheduling decides to update.

Later, add tracked ranges using the Buffer Manager edit descriptors:

- range decorations map through edits like cursors/selections
- line decorations map by changed line ranges
- producers can choose `tracking = "range" | "line" | "manual" | "clear_on_edit"`
- deleted/ambiguous ranges can either collapse, expand, or be dropped according to an affinity option

Do not block the first search decoration slice on full tracking.

## C data shape

A minimal native struct can be plain and copy-owned:

```c
typedef enum NativeDecorationKind {
  NATIVE_DECORATION_RANGE,
  NATIVE_DECORATION_LINE,
  NATIVE_DECORATION_LINE_HINT,
} NativeDecorationKind;

typedef enum NativeDecorationPlane {
  NATIVE_DECORATION_LINE_BACKGROUND,
  NATIVE_DECORATION_BACKGROUND,
  NATIVE_DECORATION_GUTTER,
  NATIVE_DECORATION_OVERVIEW,
  NATIVE_DECORATION_UNDERLINE,
  NATIVE_DECORATION_OUTLINE,
  NATIVE_DECORATION_TEXT,
  NATIVE_DECORATION_HINT,
  NATIVE_DECORATION_OVERLAY,
} NativeDecorationPlane;

typedef struct NativeDecoration {
  NativeDecorationKind kind;
  NativeDecorationPlane plane;
  size_t start_offset;
  size_t end_offset;
  size_t line_first;
  size_t line_last;
  int priority;
  unsigned int flags;
  char *style_key;
  char *text;
  size_t insertion_index;
} NativeDecoration;
```

The store should own all strings copied from Lua. It should not retain Lua references or style table pointers. Lua maps `style_key` to `style[style_key]` during drawing.

## Lua item shape rules

Validation should be strict and quiet-log useful failures:

- `producer` must be a non-empty string.
- `kind` defaults to `range` when offsets are present.
- Range decorations require numeric `start_offset` and `end_offset` with `start_offset < end_offset`.
- Line decorations require `line` or `line_first`/`line_last`.
- Unknown `kind` or `plane` is an error.
- Unknown `style` is allowed; the Lua view falls back to a plane default such as `style.search_selection`, `style.line_highlight`, or `style.accent`.
- Invalid items reject the whole producer update rather than partially installing it.

## Search vertical slice

The first producer is `search.results` plus `search.active`.

Expected behavior:

- Search UI computes visible/global literal matches for a native Buffer.
- It installs all result ranges as `plane = "background"`, `style = "search_selection"`, `priority = 100`, `clear_on_edit = true`.
- The active match can either be a separate `search.active` outline decoration or a flag on one result. Prefer a separate producer set because it changes more frequently.
- `NativeEditorView` queries background/outline decorations for the visible byte range and draws them by line alongside existing selection and syntax code.
- Closing/hiding the search UI clears search producers for the relevant Buffer.

Initial Lua example:

```lua
local items = {}
local start = 0
while true do
  local a, b = buffer:find_literal(query, start, { case_sensitive = case_sensitive })
  if not a then break end
  items[#items + 1] = {
    kind = "range",
    start_offset = a,
    end_offset = b,
    plane = "background",
    style = "search_selection",
    priority = 100,
  }
  start = math.max(b, a + 1)
end
buffer:set_decorations("search.results", items, { clear_on_edit = true })
```

## Git diff vertical slice after search

Git diff should become a Lua producer backed by native line/gutter/overview decorations:

- Lua owns git process orchestration and diff parsing.
- Native Buffer stores line markers and exposes visible queries.
- The view draws gutter bars and overview marks from generic decoration planes.
- Old `DocView` monkey-patching in `gitdiff_highlight` can be archived once the native producer is complete.

## Testing plan

Runtime/native Lua tests:

- setting/querying/clearing range decorations
- replacing one producer leaves another producer intact
- invalid item rejects the whole replacement
- query by byte range only returns intersecting range decorations
- deterministic sort by plane/priority/source/anchor
- `clear_on_edit` decorations disappear after an edit and after undo/redo snap

UI Lua tests while the view remains Lua-hosted:

- visible range query is used for native search highlights
- range decorations produce expected line rects for single-line and multi-line ranges
- active search outline can update without replacing all result highlights

Native tests later:

- range tracking through edits, once implemented
- line tracking through edits, once implemented

## Migration notes

Existing one-off native view helpers should be migrated only when there is a clear producer boundary:

- selection occurrence highlights -> view-local or Buffer decorations produced from current selection
- bracket match -> likely view-local active decorations, later Tree-sitter-aware native producer
- whitespace/indent/column guides -> view-local decoration-like helpers first; native only if performance or shared APIs require it
- diagnostics/git/search -> Buffer decorations

The important rule is that new visual features should not add new `NativeEditorView:draw_*` islands unless they are temporary adapters over the shared decoration query path.
